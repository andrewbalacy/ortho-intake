# OrthoIntake

**[Live Demo →](https://ortho-intake.vercel.app/)**

A clinical intake dashboard built on live FHIR R4 data, designed around the workflow of a walk-in orthopedic clinic. Rather than recreating an EHR, OrthoIntake focuses on one specific problem: reducing the time a front-desk or triage staff member spends navigating a chart before a patient is seen.

Built as the health IT counterpart to [Slate](https://github.com/andrewbalacy/slate) — where Slate demonstrates general product engineering, OrthoIntake demonstrates healthcare interoperability and clinical workflow design specifically, informed by 5+ years working the floor at an orthopedic walk-in clinic.

## Why this exists

Most "healthcare" portfolio projects are CRUD apps with a stethoscope icon. OrthoIntake works against a real FHIR R4 API using the actual resource types and data shapes a health IT analyst role requires day one: `Patient`, `Condition`, `AllergyIntolerance`, `Encounter`.

The intake flow is modeled on a real walk-in front desk — a waiting-room list, a chart pulled per patient, and a pre-intake briefing that surfaces what matters before a clinician opens the full chart. That's a judgment call currently made by a person scanning a screen; this is what it looks like automated without replacing that judgment.

## What it does

- **Patient Intake** — pulls a patient's demographics, active conditions, allergies, and encounter history in real time, with a pre-intake briefing that summarizes the most relevant facts at a glance
- **Orthopedic relevance engine** — automatically flags patients with prior orthopedic-relevant history directly in the waiting-room list, before a chart is even opened
- **FHIR Explorer** — a raw-resource view showing exactly what's coming back from the FHIR server for each patient, side by side with what OrthoIntake extracts from it. This exists deliberately: the interoperability layer is visible, not hidden behind a polished UI
- **Encounter Timeline** — chronological encounter history per patient, correctly ordered by date (see "A real bug, fixed" below)

Every summary card carries the same disclaimer pattern throughout: this organizes chart information, it does not make clinical decisions. ("Chart context only — not clinical guidance," "Confirm with patient at intake.")

## Data source

Reads from the [SMART Health IT public R4 sandbox](https://r4.smarthealthit.org), a free, no-auth, read-only FHIR server maintained by the SMART Health IT project. All patient data is synthetic, generated by [Synthea](https://synthetichealth.github.io/synthea/) — no real PHI touches this project at any point.

All fetching happens client-side via unauthenticated `GET` requests. This does not implement the full SMART App Launch / OAuth2 flow (patient context handoff from an EHR session) — that's a deliberate scope decision for v1, and the natural next step (see below).

## A real bug, fixed

Two different views (Patient Intake and FHIR Explorer) were independently querying the Encounter endpoint with different `_count` values against a FHIR server that returns bundle entries in arbitrary, non-date-sorted order. That meant each view could sort a different pool of encounters and land on a different "most recent" encounter for the same patient — producing inconsistent Reason for Visit and appointment type between the two screens.

Fixed by extracting shared `getEncounterDate` / `hasPeriod` functions as a single source of truth for encounter ordering, and standardizing both fetch paths on the same `_count`. It's a small bug, but a real one — the kind that only shows up when you're actually testing across views instead of just shipping one screen and calling it done.

## Stack

Next.js 15 (App Router) · TypeScript · Tailwind CSS · deployed on Vercel

## Running locally

```bash
npm install
npm run dev
```

Open `http://localhost:3000`. No environment variables or API keys needed — the FHIR sandbox is public.

## What's next

- Full SMART App Launch (OAuth2 + launch context) against a sandbox like `launch.smarthealthit.org`, so the app can be "launched" from a simulated EHR session with patient context already provided — the pattern real EHR-embedded apps use
- Replace keyword-based orthopedic relevance matching with a proper SNOMED CT value-set match
- Add `Observation` (vitals) and `MedicationRequest` to the chart view
- Acuity flagging: distinguish recent/acute orthopedic conditions from long-standing chronic ones in the waiting room, not just "ortho-relevant vs. not"

## Disclaimer

This is a portfolio demonstration project using public synthetic data. It is not connected to any production EHR, does not handle real patient data, and is not a certified clinical tool.
